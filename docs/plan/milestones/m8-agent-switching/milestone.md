# m8: Agent Switching Without Data Loss

**Status: Complete**

Layered switching, centralized runtime ownership, and legacy-layout guardrails shipped.

Make it easy to switch between supported agents without losing:

- Agent state volumes (credentials, history, tool state)
- Project policy customizations
- Compose customizations

## Problem

Current workflow is destructive for users who customize their setup:

- `agentbox init` recopies templates and overwrites compose/policy files.
- Users can keep only one active generated setup at a time.
- Trying another agent often means re-running init and manually re-applying edits.

This creates friction for experimentation and increases risk of accidental config loss.

## Goals

- Add a first-class `agentbox switch --agent <name>` workflow.
- Preserve all existing agent volumes when switching.
- Preserve user customizations across switches by separating managed files from user-owned files.
- Keep devcontainer support additive via a thin `.devcontainer/` IDE shim backed by `.agent-sandbox` runtime files.
- Define a data model that is concurrency-ready (can support multiple running agents later) while keeping a simple single-active UX now.

## Non-Goals

- Full multi-target orchestration UX in this milestone (no first-class "run many targets concurrently" commands yet).
- Unifying agent-specific auth/config semantics (`.claude`, `.codex`, `.copilot`) into one shared home.
- Solving arbitrary malformed/manual compose edits with zero constraints.
- Automatic migration tooling for legacy layouts.

## Proposed Design

### 1) Target identity and UX model

Define runtime target as `{agent}`.

- Examples: `codex`, `claude`
- `switch` sets the active agent used by default command paths.

UX in m8:

- Single-active agent is the default mental model.
- Internally store agent identity in a way that can support concurrent targets later.

### 2) Runtime ownership split

Treat runtime control consistently for CLI workflows, and keep devcontainer support as an IDE-facing shim over the same runtime ownership model:

- CLI runtime: agentbox-managed runtime (`up`, `down`, `exec`, `logs` supported via compose backend).
- Devcontainer runtime: IDE/devcontainer-managed, discovered through `.devcontainer/devcontainer.json` but backed by `.agent-sandbox` compose and policy files.

Implication:

- `agentbox switch` does not toggle CLI vs devcontainer.
- `agentbox switch` updates the active-agent-backed runtime files and regenerates `.devcontainer/devcontainer.json` to match the selected agent.
- Users who want devcontainer workflow use IDE reopen/rebuild flow after a switch when needed.

### 3) Compose file ownership and layering

Introduce explicit ownership boundaries:

- Managed files (agentbox-owned):
  - `.agent-sandbox/compose/base.yml`
  - `.agent-sandbox/compose/agent.<agent>.yml`
  - `.agent-sandbox/compose/mode.devcontainer.yml`
- User-owned files (never overwritten):
  - `.agent-sandbox/compose/user.override.yml` (project-wide)
  - `.agent-sandbox/compose/user.agent.<agent>.override.yml` (optional)
- IDE-facing devcontainer files:
  - `.devcontainer/devcontainer.json`
  - `.devcontainer/devcontainer.user.json` (optional)

Why this split: preserving arbitrary user edits in one generated file is brittle, but splitting runtime state between `.agent-sandbox/` and `.devcontainer/` is also confusing. Layered ownership should live in one runtime directory, with `.devcontainer/` reserved for IDE discovery.

### 4) Compose execution model (CLI workflow)

For CLI workflow and devcontainer-backed workflow, compose merge order should be deterministic:

1. base
2. agent
3. mode.devcontainer (devcontainer only)
4. user.shared
5. user.agent (if present)

This supports agent-specific customizations (e.g. `CLAUDE.md` mounts) without polluting global overrides, while keeping devcontainer-only runtime deltas small and explicit.

### 5) Policy layering model (merge at proxy runtime)

Use layered policy inputs, merged at proxy startup/runtime (not pre-generated by agentbox):

- `.agent-sandbox/user.policy.yaml` (user-owned)
- `.agent-sandbox/user.agent.<agent>.policy.yaml` (optional, user-owned)
- `.agent-sandbox/policy.devcontainer.yaml` (agentbox-managed; applied for devcontainer workflows)

Baseline agent policy remains agent-managed (template/image default).

Merge semantics:

- `services`: union with de-duplication
- `domains`: union with de-duplication
- Unknown keys: preserve/deep-merge where feasible

Rationale:

- Avoid stale generated policy artifacts.
- Makes concurrent targets easier later.
- Keeps one merge implementation path.

### 6) `agentbox switch` behavior

`agentbox switch --agent <claude|codex|copilot>`:

- Validate requested agent.
- Interactive mode:
- If no flag: prompt exactly one question (agent).
- If flag is present: do not prompt.
- Do not ask unrelated questions (project name, IDE, etc.).
- Update active agent pointer.
- Ensure required user-owned override files exist (create scaffold only if missing).
- Reconcile running services safely (`up -d`), never `down --volumes`.
- Update the centralized runtime files and regenerate `.devcontainer/devcontainer.json` to match the selected agent by default.
- Never rewrite user-owned `.agent-sandbox/` override files or `.devcontainer/devcontainer.user.json`.

### 7) Volume preservation guarantees

- Keep agent-specific named volumes (`claude-state`, `codex-state`, `copilot-state`).
- Switching changes selected agent and mounted config, not volume lifecycle.
- Existing volumes remain intact and reusable when switching back.

### 8) Breaking-change rollout (existing installs)

Use explicit breaking-change upgrade guidance:

- No automatic migration command.
- New `init`/`switch` target only layered layout.
- If legacy single-file setup is detected, fail with clear instructions:
  - Re-run `agentbox init` (new layout)
  - Manually copy custom entries to user override and shared policy files

## Alternatives Considered

### A) Keep single compose and patch in place with `yq`

- Pros: minimal file churn.
- Cons: fragile with comments, custom service structure, and non-standard edits.
- Rejected.

### B) Keep devcontainer runtime state in `.devcontainer/`

- Pros: all devcontainer-related artifacts are physically colocated.
- Cons: users have to understand runtime and policy layering across two directories, edit commands become mode-dependent, and managed devcontainer compose/policy files duplicate the layered CLI backend.
- Rejected after `m8.4` user feedback.

## Risks

- User confusion about which files are safe to edit.
- Confusion about the relationship between `.devcontainer/` IDE files and `.agent-sandbox/` runtime files.
- Policy merge surprises (expecting replacement semantics vs union).
- Future concurrent-target rollout may expose naming/identity assumptions if not designed now.

## Mitigations

- Strong file headers (`managed` vs `user-owned`).
- Keep `.devcontainer/` thin enough that it does not carry runtime policy ownership.
- `agentbox doctor` checks for missing/inconsistent target files.
- `agentbox policy render` should reuse the same merge path as proxy runtime.
- BATS coverage for switching, guardrails, and non-destructive behavior.
- Regression assertions should target effective config via `docker compose config --no-interpolate` to be layout-agnostic.

## Implementation Plan

### m8.1: Target model + switch CLI

- Add target identity model (`agent`) and active-target state.
- Add `agentbox switch` with validation.
- Add interactive prompt flow for missing `--agent` values.

### m8.2: Layered compose layout

- Refactor `init` to produce managed compose layers.
- Create user override scaffolds if absent; never overwrite user-owned override files.
- Update CLI compose backend to merge files in deterministic order.

### m8.3: Layered policy model

- Add shared/agent policy files.
- Implement merge at proxy startup/runtime.
- Ensure `policy render` and runtime use the same merge implementation.

### m8.4: Devcontainer sidecar files

- Keep CLI runtime commands fully supported.
- Add managed `.devcontainer` sidecar files (`devcontainer.json`, compose override, policy override).
- Add optional user-owned `.devcontainer` override files (`*.user.*`) as extension points.
- Sync `.devcontainer` sidecar files on `switch` by default.
- Document managed vs user-owned fields and expected workflow when using devcontainer reopen/rebuild.

### m8.5: Centralize devcontainer runtime files

- Move devcontainer compose and policy runtime files into `.agent-sandbox/`.
- Keep `.devcontainer/` as an IDE shim (`devcontainer.json` plus optional `devcontainer.user.json`).
- Reuse the shared and agent-specific user override files for devcontainer workflows instead of adding devcontainer-specific user compose/policy override files.
- Add a single managed devcontainer compose overlay and a single managed devcontainer policy layer under `.agent-sandbox/`.
- Update init, switch, edit, compose resolution, policy rendering, and docs to match the new ownership model.

### m8.6: Legacy guardrails + docs/tests

- Detect legacy single-file layout and fail fast with upgrade guidance.
- Add BATS for:
  - switch preserves custom compose edits
  - switch preserves shared/agent policy overrides
  - switch preserves volumes (no `down --volumes`)
  - switch prompt behavior (0/1 prompts based on flags)
  - invalid `--agent` errors
  - legacy-layout detection guidance
- Convert `init/regression.bats` assertions to validate effective compose output via `docker compose config --no-interpolate`.
- Keep a minimal set of raw-file assertions for ownership/scaffolding behavior that rendered config cannot verify.
- Update README and agent docs with switching workflow and `.devcontainer` sync behavior.

## Success Criteria

- User can switch `codex` -> `claude` without losing prior auth/state.
- User customizations in user-owned override files survive `init`, `switch`, and `bump`.
- Shared policy edits are not duplicated per agent by default.
- Agent-specific overrides are possible when needed.
- `.devcontainer/devcontainer.json` stays in sync with the active agent after `switch`, while runtime compose/policy state remains in `.agent-sandbox/`.
- Devcontainer workflows reuse the shared `.agent-sandbox` user override files instead of requiring separate devcontainer compose/policy user override files.
- Legacy layout is detected with clear upgrade instructions.

## Open Questions

- Should `switch` auto-reconcile CLI services (`up -d`) by default or require explicit apply?
- Should policy merge support explicit subtract/deny semantics later?
